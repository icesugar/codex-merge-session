# codex-merge-session UI 改造设计

## 背景

当前 `codex-merge-session` 已具备扫描 provider、预览合并、执行合并、尝试修复、显示结果等核心能力，但界面仍停留在调试工具状态：

- 页面缺少清晰的信息层级，控件和文本直接堆叠在一起。
- 右侧操作区主次关系不明确，按钮位置和样式都不够稳定。
- 结果反馈只是简单文本，成功、提示、错误状态没有明显区分。
- 整体留白、间距、配色和卡片结构不足，影响可读性和信任感。

本次改造只处理视觉层级、布局组织和交互呈现，不改变任何核心业务逻辑。

## 目标

1. 把页面从“调试面板感”提升为“专业桌面工具感”。
2. 让用户能快速理解页面的三件事：当前数据概况、可以执行什么、刚刚发生了什么。
3. 明确操作主次关系，让 `开始合并` 成为唯一主操作，`尝试修复` 成为辅助操作。
4. 保持现有功能和数据流不变，降低 UI 调整带来的回归风险。

## 非目标

- 不新增新的业务能力或新的修复策略。
- 不调整合并、备份、修复、索引重建等底层存储逻辑。
- 不引入复杂动画、主题切换、深色模式或自定义窗口框架。
- 不改变现有中文文案的业务含义，只做必要的界面表达优化。

## 方案对比

### 方案 A：专业工具风（采用）

特点：

- 淡暖灰背景 + 白色卡片 + 细描边。
- 页面结构清晰，强调可读性和稳定性。
- 适合当前桌面维护工具定位。

优点：

- 风险低，和现有 `egui` 组件能力匹配度高。
- 不需要复杂自绘，也能明显提升质感。
- 后续继续扩展字段和状态时不容易失控。

缺点：

- 个性表达相对克制，视觉冲击力不强。

### 方案 B：强对比控制台风

特点：

- 右侧操作区使用深色面板，主按钮高亮。

优点：

- 操作区视觉焦点强。

缺点：

- 和当前左侧数据区风格容易割裂。
- 在 `egui` 中控制得不好会更像后台 Demo。

### 方案 C：极简留白风

特点：

- 极少边框，依赖留白和字体层级组织信息。

优点：

- 干净，观感轻。

缺点：

- 当前页面信息密度不低，极简方案容易让数据和状态显得更散。

## 1. 整体视觉方向

- 页面背景使用非常淡的暖灰色，降低纯白界面的生硬感。
- 所有核心区域使用统一白色卡片承载。
- 卡片统一使用：
  - 12px 左右圆角
  - 轻描边
  - 明显但克制的内边距
- 字体层级：
  - 页面标题：深灰、大号、半粗体
  - 卡片标题：深灰、中号、半粗体
  - 说明文字：中灰
  - 辅助信息：更浅灰

## 2. 页面结构

页面分为三块：

1. 顶部摘要栏
2. 中部双卡片主区
3. 底部结果状态卡

### 2.1 顶部摘要栏

顶部不再做成简单文本堆叠，而是做成轻量信息头：

- 主标题：`codex-merge-session`
- 次要信息：
  - `Codex 根目录`
  - `当前配置 provider`

要求：

- 次要信息使用更弱的颜色和更小字号。
- 顶部区域只承担“我现在在哪个数据目录上操作”的说明，不再承载任何操作按钮。

### 2.2 中部双卡片主区

中部保持左右两列，但改成两个等高主卡片：

- 左侧：`Provider 概览`
- 右侧：`会话操作`

要求：

- 列间距明显，卡片之间有呼吸感。
- 小窗口宽度下仍保持可读，不出现按钮和文本互相挤压。

### 2.3 底部结果状态卡

底部横向独立一张卡片，用于承载：

- 错误信息
- 最近一次执行结果
- 最近一次修复结果

要求：

- 通过状态底色区分不同结果。
- 即使没有执行结果，也要显示清晰的空状态提示。

## 3. 左侧卡片：Provider 概览

### 3.1 头部

卡片标题：`Provider 概览`

标题下补一行摘要：

- `已发现 N 个 provider`

### 3.2 列表呈现

保留当前的四列信息：

- `Provider`
- `活跃`
- `归档`
- `最近更新时间`

展示规则：

- 表头弱化处理，不要比标题更抢眼。
- 行高提高，避免内容拥挤。
- 使用斑马纹或轻背景区分相邻行。
- 数字列右对齐，提高扫读效率。
- 当前目标 provider 所在行做轻微高亮，帮助用户建立“左边看概况，右边做操作”的联系。

### 3.3 空状态

当扫描结果为空时，显示一条卡片内提示：

- `尚未加载到 provider 数据。`

要求：

- 空状态保持版式完整，不出现卡片塌缩。

## 4. 右侧卡片：会话操作

### 4.1 头部操作区

标题：`会话操作`

标题右侧放一组纵向操作按钮，右对齐：

- 主按钮：`开始合并`
- 次按钮：`尝试修复`

按钮策略：

- `开始合并`
  - 可执行时：深蓝绿色实心按钮
  - 禁用时：低饱和灰蓝按钮
  - 比次按钮更高、更醒目
- `尝试修复`
  - 白底
  - 蓝灰描边
  - 作为辅助操作，不与主按钮抢视觉焦点

### 4.2 表单区

操作按钮下方是清晰的表单区域，包含：

- `目标 provider` 标签
- provider 下拉框
- `重新扫描`
- `刷新预览`

要求：

- 表单区和按钮区视觉分层明确。
- 下拉框和轻按钮不要直接贴边。

### 4.3 预览信息区

当前预览信息改成更清晰的信息块，至少展示：

- 待迁移线程数
- 待迁移文件数
- 来源 provider
- 备份目录

展示策略：

- 计数信息更突出。
- 长路径允许折行或弱化展示，但不能破坏卡片版式。

### 4.4 提示区

没有可合并会话时，不再只显示普通文本，而是展示一条浅色提示条：

- `当前没有可合并的会话，或预览尚未就绪。`

要求：

- 提示条视觉上区别于错误信息。

## 5. 底部卡片：执行结果与错误

标题：`执行结果与错误`

状态分为三类：

1. 成功
2. 提示 / 无需处理
3. 错误

展示策略：

- 成功：浅蓝绿色背景
- 提示：浅灰蓝或浅黄色背景
- 错误：浅红背景 + 深红文字

文案要求：

- 中文直白、可读，避免暴露内部字段名。
- 优先表达结果，再表达细节。

示例：

- `已完成本次合并，成功处理 90 条会话，更新了 90 个会话文件。`
- `已完成一次数据修复，路径和会话索引已同步刷新。`
- `当前没有需要合并的数据。`

## 6. 确认合并弹窗

保留居中弹窗，但做统一风格收敛：

- 标题：`确认合并`
- 主体内容包含：
  - 目标 provider
  - 将迁移的线程数
  - 涉及的源 provider 数量
  - 备份目录
- 底部按钮：
  - 主按钮：`确认执行`
  - 次按钮：`取消`

要求：

- 弹窗宽度稳定，内容留白充足。
- 主次按钮样式和右侧操作区保持一致。

## 7. 实现边界

本次允许改动的范围：

- `src/app.rs` 中的页面布局、按钮样式、状态呈现
- 若有必要，可在 `src/main.rs` 中补充全局视觉配置，例如背景或间距相关初始化

本次不改动：

- `CodexStore` 及其底层读写逻辑
- 合并、修复、回滚、备份、索引重建算法
- 图标、字体资源和窗口行为

## 8. 验证策略

### 8.1 代码级验证

- 保留现有按钮样式相关测试并按新样式更新断言。
- 为新的布局辅助函数保留测试，确保按钮组继续右对齐。
- 结果文案测试继续保留，避免 UI 调整时误改业务提示。

### 8.2 构建验证

- 运行 `cargo test`
- 运行 `cargo build --release`
- 若默认 release 因 exe 占用无法覆盖，则额外运行 `CARGO_TARGET_DIR=target-verify cargo build --release`

### 8.3 体验验证

- 启动程序检查窗口在默认大小下的布局是否稳定。
- 检查以下状态：
  - 无可合并会话
  - 有可合并会话
  - 执行成功
  - 修复成功
  - 错误提示

## 9. 风险与控制

### 风险

- `egui` 自定义能力有限，过度追求复杂视觉会带来实现成本和布局不稳定。
- 路径文本较长，容易挤压右侧卡片。
- 小窗口宽度下按钮和下拉框可能发生布局竞争。

### 控制策略

- 保持卡片式布局，不引入复杂自绘。
- 优先通过层级、间距、描边和颜色改善观感，而不是堆砌装饰。
- 对长文本使用折行或较弱视觉权重，避免其成为主视觉元素。

## 10. 实现顺序建议

1. 先整理整体页面结构和卡片容器。
2. 再重做右侧按钮区和表单区。
3. 接着重做左侧列表和底部状态卡。
4. 最后统一调整弹窗和配色细节。
